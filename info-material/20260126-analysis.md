## 1. Bug Discovery in Original mask_audio.py

### Root Cause

The original `mask_audio.py` had a critical indentation bug where `temp_results` was cleared **every chunk** instead of only after saving:

```python
if len(temp_results[...]) >= 25000 or chunk_end == total_samples:
    # save logic...

# BUG: This line runs EVERY iteration, not just after saving
temp_results = {condition: [] for condition in hearing_profiles.keys()}
```

### Impact on CV16.1 Processing

| Split | Expected Samples | Actually Saved | Data Loss |
|-------|-----------------|----------------|-----------|
| train | 1,090,061 | 50,061 | **95.4%** |
| validation | 16,390 | 1,390 | **91.5%** |
| test | 16,390 | 1,390 | **91.5%** |

The processed CV16.1 datasets (normal, lfloss, hfloss) contain only ~5-8% of the original data.

## 3. Feasibility Assessment

### Option A: Process Full CV24.0 Dataset

- **Storage required:** ~2.7 TB
- **Processing time:** ~15-20 hours (estimated)
- **Memory:** Fixed script handles this with ~6 GB peak memory
- **Verdict:** Technically feasible if storage is available

### Option B: Process Same Subset Size as CV16.1 (~5%)

- **Samples:** ~59,000 (train) + ~800 (dev) + ~800 (test)
- **Storage required:** ~130 GB (similar to current CV16.1 processed)
- **Verdict:** Feasible, but wastes 95% of available data

### Option C: Process Meaningful Subset (e.g., 20%)

- **Samples:** ~230,000 (train) + ~3,300 (dev/test each)
- **Storage required:** ~540 GB total
- **Verdict:** Good balance of data coverage and storage

### Option D: Store Only Log-Mel Spectrograms (Skip Intermediate Audio)

Instead of:
```
mask_audio.py -> HuggingFace Dataset -> DataSet2LogMel.py -> Training
```

Do:
```
mask_audio.py -> directly to log-mel spectrograms -> Training
```

- Log-mel spectrograms are much smaller (~50 KB per sample vs 2.2 MB)
- **Storage for full CV24.0:** ~180 GB total (vs 2.7 TB)
- **Verdict:** Most storage-efficient, but requires pipeline modification

## 5. Conclusion

**Processing CV24.0 is feasible**, but the storage requirements are significant due to the uncompressed audio format used by HuggingFace Datasets.

**Key decisions needed:**

1. How much storage is available for processed data?
2. Is modifying the pipeline to skip intermediate audio storage acceptable?
3. What subset size (if any) provides acceptable model performance?

The bug in the original script means your colleagues trained on only ~5% of available data. With proper processing, you have the opportunity to use 20x more training data, which could significantly improve model performance.

## 7. The Compression Problem

### Current Pipeline is Inefficient

The current pipeline decompresses audio and stores it uncompressed:

```
MP3 (compressed, ~40KB/sample)
    ↓ decompress
Float32 arrays (uncompressed, ~2.2MB/sample)
    ↓ save to Arrow
HuggingFace Dataset (2.2MB/sample, 55x expansion)
    ↓ convert
Log-mel spectrograms (for training)
```

**The fundamental issue:** HuggingFace's Arrow format stores audio as **uncompressed float32 arrays**. This is optimized for fast random access during training, but terrible for storage and sharing.

### Implications for Mozilla Upload

If the goal is to upload hearing-loss-simulated datasets to Mozilla or share with the community, the current approach is problematic:

1. Process CV24.0 → 2.7 TB of uncompressed Arrow data
2. Load and re-compress each sample to MP3/FLAC
3. Upload the compressed version (~100 GB)

This is wasteful - decompressing then re-compressing.

### Better Approaches

| Approach | Description | Output Size (full CV24) | Use Case |
|----------|-------------|------------------------|----------|
| **A: Direct to spectrograms** | Skip intermediate audio storage, go MP3 → apply mask → spectrograms | ~180 GB | Local training only |
| **B: Compressed audio output** | Apply masks, save as MP3/FLAC instead of Arrow | ~100-150 GB | Sharing with community |
| **C: On-the-fly processing** | Don't store intermediate, process during training | 0 GB extra | Memory-rich environments |

### Recommended Pipeline by Goal

**Goal 1: Train Whisper models locally**
```
MP3 → [apply hearing loss mask] → [generate spectrograms] → Save
```
- Skip intermediate audio storage entirely
- Output: ~180 GB of spectrograms (vs 2.7 TB current approach)

**Goal 2: Share datasets with Mozilla/community**
```
MP3 → [apply hearing loss mask] → [re-compress to MP3/FLAC] → Save/Upload
```
- Keep audio compressed throughout
- Output: ~100-150 GB of compressed audio

**Goal 3: Both training and sharing**
```
MP3 → [apply hearing loss mask] → [save as compressed FLAC] → Save
                                         ↓
                            [generate spectrograms for training]
```
- Store compressed audio for sharing
- Generate spectrograms on-demand for training

### Action Required

The current `mask_audio.py` saves to HuggingFace Arrow format (uncompressed). To achieve practical storage sizes, the pipeline needs modification to either:

1. Output compressed audio (MP3/FLAC) instead of Arrow, or
2. Skip audio storage and output spectrograms directly

This is a design decision that affects the entire downstream pipeline.

## 9. Recommended Approach: FLAC + TSV Output Format

### Project Goal (as stated by David)

The primary goal is to:
1. Generate **hfloss** and **lfloss** filtered versions of CV24.0
2. **Share with Mozilla/community** (upload to Mozilla Common Voice or similar)
3. Maintain compatibility with the rest of the repository for local training

### Recommended Solution: Mozilla-Compatible Output

Instead of HuggingFace Arrow format, output **FLAC files + TSV metadata** that mirrors Mozilla's Common Voice format:

```
output/
├── hfloss/
│   ├── clips/
│   │   ├── common_voice_en_12345.flac
│   │   ├── common_voice_en_12346.flac
│   │   └── ...
│   ├── train.tsv
│   ├── dev.tsv
│   └── test.tsv
├── lfloss/
│   ├── clips/
│   │   └── ...
│   ├── train.tsv
│   ├── dev.tsv
│   └── test.tsv
```

### Why FLAC Instead of MP3?

| Format | Compression | Quality | Use Case |
|--------|-------------|---------|----------|
| MP3 | Lossy | Additional quality loss | Not recommended (double lossy compression) |
| FLAC | Lossless | No additional loss | Recommended for research datasets |

**Storage estimate:** ~100-150 GB for full CV24.0 (vs 2.7 TB for Arrow format)

### Benefits of This Approach

1. **Sharable with Mozilla** - Mirrors their exact format
2. **Storage efficient** - FLAC is ~100-150 GB vs 2.7 TB Arrow
3. **No re-compression needed** - Direct output, no intermediate storage
4. **Compatible with local training** - `DataSet2LogMel.py` can be modified to read FLAC+TSV

### Implementation Feasibility

**Estimated effort:** Moderate (6-8 hours)

**Key insight:** `scripts/utils/cv24_loader.py` already implements TSV+audio loading for CV24. This can be adapted:
1. Copy `cv24_loader.py` → `flac_tsv_loader.py`
2. Modify `mask_audio.py` to output FLAC files instead of Arrow
3. Update `DataSet2LogMel.py` to use the new loader

**Code already available:**
- `cv24_loader.py` line 102-106: Audio loading with librosa (handles FLAC transparently)
- `cv24_loader.py` line 137-153: Multiprocessing-safe `.copy()` method
- `cv24_loader.py` line 246-304: TSV parsing and dataset creation

### Changes Required to mask_audio.py

1. **Remove:** Arrow/HuggingFace saving logic, chunk merging
2. **Add:** FLAC file writing using `soundfile.write()`
3. **Add:** TSV metadata generation (path, sentence, client_id, etc.)
4. **Simplify:** No more hierarchical merge needed - just write files directly

This would actually **simplify** the code by removing the complex chunk merging logic.
